# delivery-system

QAエンジニアとしてのテスト自動化スキルを示すためのポートフォリオプロジェクト。

果物ECサイトを題材に、Playwright による E2E テスト自動化と、GitHub Actions による CI/CD パイプラインを構築しています。意図的に仕込んだバグを検知するテスト設計と、実際に遭遇した flaky テストの安定化を通じて、実践的なテスト自動化の考え方を反映しました。

## アプリ概要

果物を注文できる EC サイト。以下の5画面で構成されています。

- 会員登録
- ログイン
- 商品一覧（カート機能）
- 注文確認（小計・送料・合計の計算）
- 注文履歴

データベースは users / products / orders / order_items の4テーブルを外部キーで連結しています。

---

## このプロジェクトの狙い

テスト自動化スキルを示すため、アプリに**意図的なバグを2種類**仕込み、それを検知するテストを設計しています。

### バグ①：注文履歴の上書き
追加で注文すると、過去の注文履歴が消えてしまうバグ。「複数回注文しても全ての履歴が保持される」という仕様を確認するシナリオで検知します。

### バグ②：送料の境界値
「3000円以上で送料無料」のはずが、判定が `> 3000` になっており、3000円ちょうどだと送料がかかるバグ。境界値テストで検知します。

> テストは「バグを探すため」ではなく「仕様を満たすかを確認するため」に設計しています。結果としてバグが見つかる、という考え方を重視しました。

## テスト設計

### 自動化したシナリオ

| シナリオ | 目的 |
|----------|------|
| 購入フロー正常系 | 主要機能の回帰テスト |
| 注文履歴の累積確認 | バグ①の検知 |
| 送料境界値確認 | バグ②の検知 |
| メール重複エラー | 異常系の確認 |

回帰価値・バグの起きやすさ・動作の安定性を基準に、価値の高いシナリオを厳選しました。「すべてを自動化しない」という判断もテスト設計の一部と考えています。

### Page Object Model (POM)

画面ごとにクラスを分け、画面の操作と要素の知識をそのクラスに閉じ込めています。

- 検証（期待値チェック）はテスト側に置き、POM は「操作」と「要素の提供」に徹する
- 個別入力メソッドとまとめメソッドを両立させ、まとめメソッドは個別メソッドを呼ぶことで二重管理を回避
- メソッドの内部実装を変更しても、テストコード側は影響を受けない構造

### テスタビリティ

検証対象の要素に `data-testid` を付与し、CSS クラスや表示文字列に依存しないセレクタを実現しています。デザイン変更に強いテストを意識しました。

### 待機戦略

固定時間待機（`waitForTimeout`）を避け、`toHaveCount` / `toHaveValue` / `toHaveText` などの待機付きアサーションを使用しています。条件が満たされるまで待つことで、不安定さを抑えています。

## flaky テストへの対応

開発の過程で実際に複数の flaky テストに遭遇し、原因特定と安定化を行いました。詳細は [docs/flaky-test-handling.md](docs/flaky-test-handling.md) に記録しています。

主な対応：
- 並列実行時のデータ衝突 → テストデータのユニーク化で独立性を確保
- 描画前の要素取得 → 待機付きアサーションへの変更
- 画面操作の積み重ねによる不安定 → 前提条件を DB 直接作成に変更し、画面操作を最小化

待機を足す対症療法に限界を感じた際、テストの設計そのものを見直すことで根本解決した経験を反映しています。

## CI/CD (GitHub Actions)

main ブランチへの push 時に、3つのブラウザ（Chromium / Firefox / WebKit）でテストを自動実行します。

実行手順：
1. リポジトリのチェックアウト
2. Node.js のセットアップ
3. 依存パッケージのインストール（`npm ci`）
4. Playwright ブラウザのインストール
5. データベースのシード
6. 本番ビルド（`npm run build`）
7. テスト実行（本番サーバーを自動起動してテスト）

本番ビルドした状態でテストすることで、より実環境に近い検証を行っています。

> 現在、仕込んだバグ②③を検知するため CI は意図的に失敗させています。これは「回帰テストがバグを継続的に検知できている」状態を示しています。

## ローカルでの実行

```bash
# 依存インストール
npm ci

# Playwright ブラウザのインストール
npx playwright install

# データベースのシード
npm run seed

# テスト実行（本番サーバーを自動起動）
npx playwright test

# テストレポートの表示
npx playwright show-report
```
