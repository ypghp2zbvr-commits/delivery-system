# Flakyテスト対応の記録

## このドキュメントについて

本ドキュメントは、delivery-system のテスト自動化を構築する過程で実際に遭遇した「flakyテスト（実行のたびに結果が変わる不安定なテスト）」と、その原因特定・解決の記録である。

E2Eテストの自動化では、画面の描画タイミングやテストデータの競合など、実装コードには現れない不安定さに直面することが多い。本記録は、そうした問題にどう向き合い、どう解決したかを示すものである。

---

## flaky事例1：メールアドレス重複確認テストの不安定を解消

### 症状

メールアドレスの重複登録を確認するため、以下の流れでテストを作成した。

1. 登録画面でデータを登録
2. ログイン画面に遷移
3. ログイン画面から登録画面に戻る
4. 登録画面で再度同じデータを入力
5. 登録ボタンを押下し、メールアドレス重複エラーが発生することを確認

3ブラウザ（Chromium / Firefox / WebKit）で並列実行していたが、2回目の登録画面で、1つ以上のブラウザで「項目未入力エラー」が発生した。想定していた「メールアドレス重複エラー」とは異なるエラーだった。

### 原因

画面遷移の直後、Reactコンポーネントの再マウント（state初期化）が完了する前に、テストの入力処理が走っていた。入力した直後に再マウントが完走し、入力値がリセットされてしまうという、タイミングの競合が原因だった。

結果として、submit時にはフォームが空の状態になり、「項目未入力エラー」が発生していた。

### 対処

最初は、入力値が反映されたことを確認するため、各フィールドの入力時に `toHaveValue` を追加した。しかし、2回目の登録で項目未入力エラーが解消しなかった。

`toHaveValue` は「入力した値が反映されたか」を入力後に確認する手段だが、その確認を通過した後に再マウントによるリセットが起きるため、根本解決にならなかった。

ここで「問題は待機にあるのではない」と切り分け、テストの設計そのものを見直した。

当初は画面遷移が多く（登録→ログイン→登録）、それがエラーの温床になっていた。そこで、テストの前提条件である「1回目の登録」を画面操作ではなくDBへの直接INSERT（`createTestUser`）に変更し、テストは「2回目の登録でエラーになること」の確認だけに集中させた。

これにより画面操作が大幅に減り、想定通りの結果が安定して得られるようになった。

### 学び

- フォームの表示を待っても、再マウントによって入力値が消えることがあるという経験を得た。
- 待機（対症療法）を足し続けても解決しない場合、テストの設計そのものを疑うべきだと学んだ。
- 「確認したいことだけを画面操作し、前提条件はDB/APIで直接用意する」というテスト自動化の原則を、実体験を通して理解した。
- AIに相談しながら問題解決を進める中で、エラーや詰まっている状況をどう言語化すればAIが正しく理解できるか、そしてAIの提案にどう取捨選択して対応するかという思考力が深まった。

## flaky事例2：並列実行でのメールアドレス衝突

### 症状

並列実行を行うと、2番目以降に実行されるブラウザでメールアドレス重複エラーが表示され、テストが失敗した。

### 原因

メールアドレスの重複登録はアプリの仕様上できない制約になっている。当初、固定の同じメールアドレスを使ってテストを構築していたため、1番目のブラウザが登録に成功すると、2番目以降のブラウザは「既に登録済み」となりテストを実行できなかった。

### 対処

ユニークな値を生成する `createTestEmail` を作成し、実行のたびに異なるメールアドレスで登録するよう修正した。

タイムスタンプとランダム値を組み合わせることで、ブラウザ間でメールアドレスが衝突しなくなり、並列実行が可能になった。

### 学び

- テストデータが他のテストと衝突しないよう、テストの独立性を保つことの重要性を学んだ。
- 並列実行のメリット（高速化）を活かすには、各テストが互いに干渉しない設計が必要だと理解した。
- ユニーク値の生成方法と、それが必要になる状況（並列実行・データ競合）を学んだ。

---

## まとめ

flakyテストへの対応を通じて、以下を実感した。

- E2Eテストの不安定さの多くは「タイミング」と「テストデータの競合」に起因する。
- 待機を足す対症療法には限界があり、テストの設計（前提条件の作り方・データの独立性）から見直すことが根本解決につながる。
- 安定したテストは、実行のたびに同じ結果を返す。「たまたま通る」状態を放置せず、複数回実行で安定性を検証することが重要である。
